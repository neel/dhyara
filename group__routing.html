Detailed Description

Dhyara Routing module is responsible for maintaining the distributed routing table across the network.

Objective of the routing module is to route a packet from the source to destination via the best intermediate nodes. Each node \(X_{i} \in X\) maintains a local view of the entire network via its routing table \(R_{i}\). Every node is aware of all other nodes in the networks, but not all routes in the network. The local view keeps track of the best neighbouring node to navigate a packet to a non-neighbouring destination. The intermediate nodes after the first hop is not visible in that local view. However that is visible to the best intermediate node's local view.

Routing Table

Each node \(X_{i} \in X\) maintains a Routing table ( \( R_{i} \)) of the following form which is gradually built up by the protocol.
Each row of this table denotes a route to a destination node (first column) via an intermediate nde (second column) and its corresponding delay. The protocol ensures that \(\forall X_{i}\in X\) is aware about existance of all other nodes in \(X \setminus X_{i}\). For all neighbours \(X_{j} \in Neibourhood(X_{i})\) \(\exists u \in R_{i}\;s.t.Intermediate(u) = 0\) For all non-neighbours \(X_{k} \not\in Neibourhood(X_{i})\) \(\exists u \in R_{i}\;s.t.Intermediate(u) \in Neibourhood(X_{i})\) The routing table may contain multiple routes for the same distination (with different intermediate nodes)

\[ R_{i} = \begin{cases} (X_{j}, 0, \delta_{j,0}) & \forall X_{j} \in Neibourhood(X_{i}) \\ (X_{k}, X_{j}, \delta_{j,0}) & \forall X_{k} \not\in Neibourhood(X_{i}) \;s.t.\; X_{j} \in Neibourhood(X_{i}) \end{cases} \]

  • 0 is used as an intermediate node to depict direct communication (not via any intermediate node).
  • Delay \(\delta_{x,y}\) is the expected delay of sending to \(X_x\) via \(X_y\) from \(X_{i}\)

Based on the above mentioned table a best route \(Best(X_{t}, i) \in R_{i}\) is calculated in \(X_{i}\) for each destination \(t\) (w.r.t. source \(X_{i}\))

\[ Best(X_{t}, i) = u \;s.t.\; Delay(u) \leq Delay(v)\; \forall\,v\,\in\,R_{i} \]

While sending data to \(X_{t}\) the best route is looked for the intermediate node.

Neighbourhood Synchronization

Each node \(X_{i}\) broadcasts a Beacon at a fixed interval. The Beacon contains Time at \(X_{i}\) ( \(T_{i}\)) at the time of sending. The Beacon is received by all \(X_{j}\) in the neighbourhood. The neighbours \(\forall X_{j} \in Neibourhood(X_{i})\) respond with an Acknowledgement, which contains the same \(T_{i}\) which was received in the Beacon. \(X_{i}\) after receiving the Acknowledgement from \(X_{j}\) calculates \(\delta_{j,0}\) as \(T_{i}/2\) and adds that as a route in \(R_{i}\).

If any operation on the routing table \(R_{i}\) alters/creates a best route or the delay of a best route for a destination \(X_{t}\) then an Advertisement is sent to \(X_{j}\,\forall X_{j} \in Neibourhood(X_{i})\) from \(X_{i}\). The Advertisement contains the destination node \(X_{t}\) and \(\delta_{t,i}\) the minimum delay to reach \(X_{t}\) from \(X_{i}\). However no Advertisement is sent to \(X_{j}\,\in Neibourhood(X_{i})\) if \(X_{j} = X_{t}\). The best intermediate node w.r.t. \(X_{i}\) (which may be 0 also) is not advertised. If the best vector is not altered by the updatation of routing table then no Advertisement packet is sent. However if \(\exists X_{k} ∊ X\) for which no broadcast packet has been sent for a large amount of time then an Advertisement regarding \(X_{k}\) is sent even if there is not change in the best vector.

Route Depreciation

If a node \(X_{i}\) doesn't receive any beacon from node \(X_{j}\) for a fixed amount of time, the delay of all routes involved with \(X_{j}\) is multiplied with a factor. Then the routing table update follows the above mentioned Neighbourhood Synchronization.
